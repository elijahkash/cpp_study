Привожу пример компонента из одного моего учебного проекта (над которым я в данный момент работаю).
Сам проект пока далек от стадии готовности и абсолютно не оформлен и не задокументирован.
Однако данный компонент практически готов.

- Классы Color и LinearSize вспомогательные и особого внимания не заслуживают.

- Классы SurfaceConfig, SurfaceOwner, SurfaceEditor реализуют сущность 'поверхность', имеют следующие роли:
  - SurfaceConfig - хранит только конфигурацию для поверхности
  - SurfaceOwner - наследуется от SurfaceConfig; при создании выделяет память и хранит ее в unique_ptr (владелец);
                   при уничтожении - освобождает ее; имеет только конструктор перемещения;
                   не предоставляет доступ к памяти.
  - SurfaceEditor - наследуется от SurfaceConfig; создается только из SurfaceOwner;
                    хранит но не владеет указателем на память; предоставляет доступ к памяти;
                    константный объект только для чтения, неконстантный - и для чтения, и для записи.

- Класс PoolOfSurfaces - потокобезопасный пул поверхностей, реализует следующую логику:
  - поверхность может быть:
    - свободной
    - поверность рендеринга (одновременно - только одна)
    - готовая поверхность (уже просчитанная, но еще не экспортированная по назначению)
    - экспортирующаяся поверхность (одновременно - только одна)
  Замечание: при размере пула '2' является примером двойной буферизации.
  - кроме служебных (для настройки) реализует 2 метода, предоствляющие вызывающей стороне объект либо для рендеринга,
                                                                                                 либо для экспорта.
    - эти методы являются ожидающими: если готовых поверхностей еще нет, метод получения поверхности для экспора
      будет ожидать ее появления т.е. окончания рендеринга (на самом деле там более сложные условия, но это уже детали).
      Тоже самое для свободной и рендерящейся поверхностей.
    - методы возвращают объекты абстрактного вложенного класса WorkingSurface - наследуемого от SurfaceEditor и
                                                                                следующие RAII идиоме:
      - уничтожение возвращенного объекта класса RenderingSurface (наследуемого от WorkingSurface) означает завершение
        рендеринга и автоматически переводит поверхность из 'рендерящейся' в 'готовую'

  Таким образом PoolOfSurfaces, инкапсулируя внутри себя всю сложную потокобезапасную работу,
  предоставляет удобный интерфейс для взаимодействия.
